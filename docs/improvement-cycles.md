# TrendDeck — Persona-Driven Improvement Cycles

This log records four iterative improvement cycles run on TrendDeck, each following the loop:
**review → prioritize areas → delegate & specialize → work on improvements → review**.
Each cycle reviewed the *actual committed result* of the previous one, so regressions and half-done
work were caught the same way a real studio cadence would catch them.

## The personas

**Reviewers** (a live games company; realistic and blunt, not sycophantic)
- **Senior Game Product Manager** — owns product outcomes: end-to-end flow, missing functionality, friction, data integrity, edge cases.
- **Senior Art Director** — owns the visual/experiential quality bar: hierarchy, type, spacing, colour, consistency, the printable brief.

**Worker team**
- **Expert Delivery Lead** — turns findings into one focused, ruthlessly-prioritized milestone (P0/P1/P2) and assigns each task.
- **Expert System Experience Designer** — specs flows, states, data & interaction logic, edge cases.
- **Expert UI/UX Specialist** — specs visual design, layout, components, motion, responsive polish.
- **Expert Software Engineer** — implements the specs in the real files, syntax-checks, and commits the cycle.

## Cycle 1 — Make one real round work end-to-end and look designed

**Commit:** `0e37ab0`

### Review

*Senior Game Product Manager —* The flow looks complete on paper, but the two steps that produce the actual deliverable — real voting and the printable brief — are broken against the real store, so a studio cannot run a live round end-to-end. The sample data renders well and demos beautifully, which masks the fact that every vote a real person casts is silently dropped and manually built/imported cards never persist. This is a convincing prototype, not yet a tool I'd put in front of a studio. Fix the store-API contract mismatches and the print path before anything else; the UI polish is largely fine.

*Senior Art Director —* This build does not currently render as a designed product. The stylesheet was written against a completely different class vocabulary than the markup the app actually produces: index.html uses BEM (app-header__brand, app-nav__link, btn--primary) and every JS module invents its own names (vote-card, td-btn, brief-winner, ingest-grid, btn-primary), while css/app.css styles app-header-inner, .brand, .nav, .btn.primary, .vs-card, .versus, .td-lb-row, etc. The overlap is near-zero. The result is unstyled or default-block content across the header, Signals, Trend Cards, Vote, Dashboard and Brief views — no primary buttons, no active-nav state, broken toast, and a head-to-head arena with none of its intended styling. The information architecture and copy are thoughtful, but until markup and CSS share one naming system, there is no visual quality bar to even evaluate. This is a critical, foundational fix, not polish.

## Cycle 2 — Heal the real-round data pipeline and unbreak the delivery surfaces

**Commit:** `9b4f8b2`

### Review

*Senior Game Product Manager —* The end-to-end happy path (sample round → vote → dashboard → printable brief) now renders and persists, which is real progress. But two pillars of the "run a round across a studio" value prop are broken or hollow: the Copy share link button generates a URL that does nothing when opened, and signals created/imported on the Signals view lose their source/thumbnail data because of a field-name mismatch with the store, which then starves the Cards and Dashboard visuals. There are also data-integrity bugs in the dashboard math and a regression where the footer's import path can silently merge a live round into the sample. This is a credible demo, not yet something I'd hand to a studio to run a real round with distributed voters.

*Senior Art Director —* The Vote view and the leaderboard look genuinely designed, and the printable brief is a credible two-page artifact when reached via the Dashboard's Export button. But the Cycle-1 claim that the CSS was "reconciled to the JS-emitted class vocabulary" is not true for the Dashboard: the entire by-department breakdown, the two primary action buttons, and the example-data badge are emitted with class names that have no CSS rule, so they render as raw unstyled browser elements on a dark page. Worse, the dedicated Brief view's own Print button prints a blank page because of a print-isolation bug. There is also a dead, fully-styled parallel dashboard vocabulary left in the CSS, which is how this gap slipped through. This is not yet something I'd put in front of leadership.

<details><summary>Key findings raised</summary>

**Product Manager:**
- `critical`/functionality — **Footer / share link (index.html + store.js):** store.js implements encodeShareLink() and a matching consumeShareFromLocation() decoder, but nothing ever calls consumeShareFromLocation on boot. index.html's boot() only seeds the store and calls showView(currentRoute()); the router's currentRoute() strips '#/' and matches against ROUTES, so a pasted '#share=...' URL falls through to DEFAULT_ROUTE and the encoded round is never imported. The 'Copy share link' footer button therefore produces a link that does nothing when a colleague opens it.
- `critical`/functionality — **Signals view persistence (ingestion.js vs store.js normalizeSignal):** ingestion.makeSignal() builds signals with keys {theme, source, thumbnail, platform}. It persists via store.addSignals(null, ...), and store.normalizeSignal only reads sourceUrl/url for the link and thumbnailUrl/thumbnail/image for the image — it does NOT read 'source'. So a user-entered Source URL is dropped on save (thumbnail survives via the 'thumbnail' alias, source does not). On reload the saved signal has empty sourceUrl, and platform is recomputed from '' → 'Other'.
- `major`/functionality — **Footer Import round file (index.html) + store.importRound default merge:** The footer import calls Store.importRound(json) with no opts. When the imported file shares an id with the active round, the default branch union-merges signals/cards/votes into the existing round, and isSample is set to (existing.isSample && incoming.isSample). If a user is on the seeded sample round (same exported id) and re-imports a colleague's export of that same sample lineage, votes get merged into the sample and the EXAMPLE DATA banner can flip off while sample content remains — or conversely real votes land inside a round still flagged sample.
- `major`/functionality — **Dashboard by-department breakdown (dashboard.js computeDepartments):** voteDepartment() maps a missing department to 'Unspecified', but the voting flow stores ANON_DEPT = 'Prefer not to say' for anonymous voters. So anonymous voters appear as a department literally named 'Prefer not to say' in the breakdown grid and brief, sitting alongside real departments rather than being excluded. Separately, suppression is by voterCount < threshold where voterCount derives from sessionId; since every vote in a session shares one sessionId this is one 'voter' per session, which is reasonable, but the 'Prefer not to say' bucket is never special-cased.

**Art Director:**
- `critical`/ui — **Dashboard view — by-department breakdown:** dashboard.js (render(), ~lines 530-549) emits .td-dept-grid / .td-dept-card / .td-dept-head / .td-dept-list / .td-dept-cardlabel / .td-dept-rate, but css/app.css has NO rules for any of these classes (only .td-dept appears inside the @media print block as part of brief styling). The on-screen department breakdown therefore renders as unstyled default <div>s and an <ol> with browser default bullets/indent — no grid, no card surface, no alignment of the win-rate values.
- `critical`/ui — **Dashboard view — primary action buttons:** The dashboard's 'Export Trend Brief' and 'Download PDF' buttons are emitted with classes .td-btn and .td-btn-primary (dashboard.js lines 482-483). css/app.css only references .td-btn inside @media print to hide it — there is no on-screen rule. These two buttons render as default OS buttons (light grey, system font) against the dark UI, right next to properly-styled .btn elements in the header/footer.
- `critical`/functionality — **Brief view (#/brief) — Print / Save as PDF:** renderBrief() prints by adding body.is-printing and calling window.print() WITHOUT injecting #td-print-root (dashboard.js lines 694-698). The print CSS rule 'body.is-printing > * { display:none !important }' hides <main> (a direct body child). The follow-up rule 'body.is-printing #view-brief { display:block }' tries to reveal the brief, but #view-brief is a descendant of the now display:none <main>, and a child cannot render when its ancestor is display:none. Result: printing from the dedicated Brief view yields a blank page. The Dashboard 'Export Trend Brief' path works only because it injects #td-print-root as a body sibling.
- `major`/ui — **css/app.css — duplicate/dead dashboard vocabulary:** The CSS carries a second, fully-styled dashboard leaderboard (.leaderboard, .lb-row, .lb-thumb, .lb-rank with r1/r2/r3 medals, .lb-winrate high/mid/low, .lb-row .winbar) and a styled .dept-table — none of which dashboard.js emits. dashboard.js uses the parallel .td-lb-* / td-dept-* vocabulary instead. Roughly two full design systems for the same view coexist; the JS happens to use the half-finished one.
